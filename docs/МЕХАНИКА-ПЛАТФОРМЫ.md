# ATAMEKEN Club — Механика платформы

> Полное описание ключевых процессов платформы **atameken.club**:
> **регистрация и доступ → приём оплаты → поддержка связи**.
> Документ отражает фактическую реализацию (бэкенд Fastify + Prisma/Postgres,
> фронтенд — статические HTML на том же сервисе Render).
>
> Версия Пользовательского соглашения: **2.0**
> Базовый URL: **https://atameken.club**

---

## 0. Обзор

ATAMEKEN Club — закрытая B2B-площадка, где предприниматели находят проверенных
контрагентов. Доступ к контактам в Реестре открывается за **токены**; сложные
сделки ведутся «под ключ» с подписанием NDA/NCNDA и комиссией.

**8 типов участников (ролей):**

| Продавцы (размещают карточки) | Покупатели (открывают карточки) |
|---|---|
| Соискатель инвестиций (`SEEKER`) | Инвестор (`INVESTOR`) |
| Франчайзер (`FRANCHISER`) | Франчайзи (`FRANCHISEE`) |
| Продавец готового бизнеса (`BIZ_SELLER`) | Покупатель готового бизнеса (`BIZ_BUYER`) |
| Продавец товаров и сырья (`GOODS_SELLER`) | Покупатель товаров и сырья (`GOODS_BUYER`) |

**Денежная единица доступа:** `1 токен = $5 USD = 1 открытие развёрнутой карточки`.

**Три опоры платформы (этот документ):**
1. Регистрация и доступ
2. Приём оплаты
3. Поддержка связи

## 1. Регистрация и аутентификация

### 1.1. Путь пользователя (фронтенд `register.html`)

1. Выбирает **тип участника** (одна из 8 ролей).
2. Заполняет: ФИО, Email, Телефон (необязательно), Пароль (мин. 8 символов).
3. Проходит **анти-бот проверку** Cloudflare Turnstile (виджет в форме).
4. Нажимает «Создать аккаунт» → открывается **модалка Пользовательского
   соглашения** (click-wrap): пользователь ставит галочку «Я прочитал(а) и
   принимаю» и жмёт «Согласен и продолжить».
5. Аккаунт создаётся, пользователь сразу авторизован и перенаправлен в кабинет.

> Принятие соглашения фиксируется в БД: дата (`termsAcceptedAt`) и версия
> (`termsVersion = 2.0`). Полный текст — `/terms.html` (26 разделов).

### 1.2. Технический процесс (бэкенд `POST /auth/register`)

```
Фронт ──► POST /auth/register { email, password, role, fullName, phone,
                                 acceptedTerms:true, captchaToken }
   │
   ├─ 1. Валидация полей (zod): email/пароль 8–128/роль из списка/acceptedTerms===true
   ├─ 2. Проверка Turnstile: POST challenges.cloudflare.com/.../siteverify
   │     (fail-closed: при ошибке сети — отказ)
   ├─ 3. Проверка дубля email → 409 «Пользователь с таким email уже существует»
   ├─ 4. Хеш пароля: bcrypt, cost 12
   ├─ 5. Создание User + выпуск пары токенов
   └─ 6. Ответ 201 + установка httpOnly refresh-cookie
```

**Токены доступа (JWT-схема):**
- **Access-token** — JWT, срок жизни **15 минут**. Хранится фронтом в
  `localStorage`, передаётся в заголовке `Authorization: Bearer …`.
- **Refresh-token** — случайные 48 байт. В БД хранится только его `sha256`-хеш.
  Отдаётся клиенту в **httpOnly Secure cookie** (`path=/auth`, `SameSite=Lax`,
  срок 30 дней). JS его не видит → защита от XSS-кражи сессии.

### 1.3. Вход, обновление сессии, выход

| Эндпоинт | Назначение |
|---|---|
| `POST /auth/login` | Проверка пароля (bcrypt), выпуск новой пары токенов |
| `POST /auth/refresh` | **Ротация**: старый refresh ревокается, выпускается новый (cookie обновляется). Работает по cookie, body не нужен |
| `POST /auth/logout` | Ревок refresh-токена в БД + очистка cookie |
| `GET /auth/me` | Текущий пользователь + статус профиля |

### 1.4. Восстановление пароля

```
POST /auth/forgot-password { email }
   └─ всегда отвечает 200 (не раскрываем, существует ли email)
   └─ если email есть: генерим raw-токен, в БД пишем только sha256(token),
      срок 1 час → ссылка /reset-password.html?token=… уходит на почту (Resend)

POST /auth/reset-password { token, password }
   └─ сверяем sha256(token), проверяем срок
   └─ ставим новый пароль (bcrypt) + ревокаем ВСЕ refresh-токены (разлогин всех устройств)
```

### 1.5. Защита от перебора (rate limiting)

- **Глобально:** 100 запросов/мин с одного IP.
- **Чувствительные auth-эндпоинты** (`/register`, `/login`, `/forgot-password`,
  `/reset-password`): **5 запросов/мин с IP** → перебор паролей и enumeration
  практически невозможны. При превышении — `429` с понятным сообщением.

### 1.6. Верификация профиля (доступ к Реестру)

Базовая регистрация бесплатна, но для размещения карточки и полного доступа
участник проходит верификацию:

```
Анкета (verify.html) ──► статус DRAFT
        │ отправка
        ▼
     SUBMITTED ──► модератор (админ) проверяет 1–2 раб. дня
        ├──► VERIFIED  (карточка публикуется в Реестре)
        └──► REJECTED  (с причиной; можно исправить и отправить повторно)
```

Админ-переход разрешён только из `SUBMITTED` или `REJECTED` (state-machine guard).

## 2. Приём оплаты

### 2.1. Токен-экономика

Доступ к контактам и развёрнутым данным в Реестре оплачивается токенами:

`1 токен = $5 USD = 1 открытие развёрнутой карточки` (или 1 пункт верификации).

**Пакеты токенов** (чем больше — тем выгоднее):

| Пакет | Токенов | Цена | Скидка |
|---|---|---|---|
| 1 | 10 | $50 | — |
| 2 | 25 | $115 | 8% |
| 3 | 50 | $220 | 12% |
| 4 | 100 | $400 | 20% |

> Цены — в USD для удобства зарубежных пользователей. Для резидентов РК сумма
> пересчитывается в тенге по курсу Нацбанка (`KZT_PER_USD`, по умолчанию 470).

### 2.2. Оплата через Kaspi Pay

```
1. Пользователь жмёт «Купить» на пакете
        │
2. POST /wallet/payment/init { packageSize }
   ├─ создаётся PendingPayment (статус PENDING, срок оплаты 30 минут)
   ├─ priceKzt = priceUsd × KZT_PER_USD
   ├─ генерится короткий референс (для чека Kaspi)
   └─ ответ: { kaspiPayUrl, reference, priceKzt, tokensAmount, expiresAt, mode }
        │
3. Фронт показывает модалку с QR-кодом (Kaspi) + кнопкой «Открыть Kaspi.kz»
        │
4. Подтверждение оплаты — двумя путями:
   • REAL-режим: Kaspi шлёт webhook → POST /wallet/payment/kaspi-webhook
   • Фронт параллельно поллит GET /wallet/payment/:id/status каждые 4 сек
        │
5. При COMPLETED → токены зачисляются в кошелёк, фиксируется транзакция PURCHASE,
   баланс на дашборде обновляется.
```

**Безопасность webhook (REAL-режим):**
- Подпись запроса проверяется через **HMAC-SHA256** (`x-kaspi-signature`),
  сравнение — `timingSafeEqual` (защита от timing-атак).
- Если `KASPI_API_TOKEN` не задан — webhook **полностью отключён** (нельзя
  подделать оплату). Работает только mock-режим.

**MOCK-режим (текущий, до получения merchant-аккаунта Kaspi):**
- `POST /wallet/payment/:id/mock-complete` — кнопка «Я оплатил (mock)»
  подтверждает платёж для UAT/демо. На проде заменяется реальным webhook'ом
  после KYC у Kaspi.

**Статусы платежа:** `PENDING → COMPLETED | EXPIRED | FAILED`
(просроченные PENDING автоматически помечаются `EXPIRED`).

### 2.3. Списание токенов (атомарно)

Открытие карточки — `POST /wallet/spend/card { publicationId }`:

```
$transaction:
   updateMany TokenWallet WHERE id=… AND balance >= 1
       SET balance = balance - 1, totalSpent += 1
   если 0 строк обновлено → 402 «Недостаточно токенов»
   иначе → запись транзакции SPEND_CARD + создание PurchasedCard
```

Условие `balance >= 1` прямо в `UPDATE` исключает «двойное списание» при
одновременных запросах (race condition). Купленная карточка (`PurchasedCard`)
остаётся доступной без срока.

### 2.4. Журнал транзакций

Каждая операция фиксируется: тип (`PURCHASE`, `SPEND_CARD`, `SPEND_VERIF`,
`REFUND`, `BONUS`), сумма, баланс после, метаданные. Виден пользователю в
разделе «Пакеты токенов → История транзакций».

### 2.5. Платные услуги и комиссия

Помимо токенов, платформа предлагает услуги (через аккредитованных партнёров и
напрямую):

- **Верификации/аудиты:** верификация компании ($50), маркетинговый анализ
  ($100), аудит бизнес-плана / фин. модели / договоров (по запросу),
  верификация состава, запасов и т.д.
- **Сопровождение сделки:** комиссия **1,5% от суммы сделки**.
  *Для крупных сделок предусмотрен верхний лимит комиссии — не более $1,5 млн.*
- Резидентам Бизнес-сообщества — скидка **−10%** на все платные услуги.

> Возврат средств — только в случаях, прямо предусмотренных правом или
> отдельным соглашением (см. п.7 Пользовательского соглашения).

## 3. Поддержка связи (коммуникации)

Платформа связывает участников и поддерживает их по нескольким каналам.

### 3.1. Чат между пользователями (Сообщения)

Прямой мессенджер между сторонами Реестра — переговоры без посредников.

```
• Старт диалога: на развёрнутой карточке кнопка «✉ Написать автору»
  → POST /chat/conversations { publicationId } (создаёт/находит диалог)
• История:  GET  /chat/conversations,  GET /chat/conversations/:id/messages
• Отправка: POST /chat/messages { conversationId, text, lang }
• Real-time: WebSocket /chat/ws
```

**Безопасная WS-аутентификация (ticket-flow):**
- Сначала `POST /chat/ws-ticket` (по JWT в заголовке) выдаёт **одноразовый
  ticket** (TTL 60 сек). Подключение: `wss://…/chat/ws?ticket=…`.
- JWT **никогда не уходит в URL** → не светится в access-логах CDN.
- WS-соединение держит heartbeat (ping 30с, таймаут 60с).

**AI-переводчик:** сообщения автоматически переводятся на язык собеседника
(как в WeChat) через OpenAI — общение с предпринимателями из любых стран на
родном языке. Язык задаётся на уровне диалога; переключатель в шапке чата.

### 3.2. Служба Поддержки (закреплённый чат)

В разделе «Сообщения» всегда первым закреплён неудаляемый чат
**«Служба Поддержки»** (🛎, 24/7).

```
Пользователь пишет → POST /chat/support { text }
        └─ сообщение форвардится в Telegram-группу модераторов (через бот)
        └─ пользователю — авто-подтверждение «сообщение получено, ответим в течение часа»
```

История переписки с поддержкой хранится локально у пользователя; менеджер
отвечает на указанные при регистрации email/телефон.

### 3.3. Внешние каналы связи

| Канал | Где | Назначение |
|---|---|---|
| **WhatsApp** | Подвал всех страниц | `+7 706 515 0336` (`wa.me/77065150336`) с готовым текстом «Здравствуйте! Пишу с сайта atameken.club.» |
| **Telegram** | Подвал / поддержка | `@atameken_support` + закрытая группа модераторов |
| **Email** | Подвал / письма | `info@atameken.club` (общий), `support@atameken.club` (поддержка) |
| **Транзакционные письма** | Resend | Восстановление пароля, уведомления |

### 3.4. Сопровождение сделки (документооборот)

Для безопасного закрытия сделки платформа ведёт её «под ключ»:

```
Создание сделки → POST /deals  (можно «делим комиссию пополам с контрагентом»)
        │
Подписание 3 документов онлайн (по очереди):
   1. NDA   — соглашение о неразглашении (взаимодействие + 3 года)
   2. NCNDA — ненарушение каналов (межд. стандарт ICC)
   3. Договор о сопровождении за комиссию 1,5%
        │
Каждое подписание: POST /deals/:id/sign — фиксируется с IP и временем
   (юридически = электронный акцепт). Когда подписаны все 3 → сделка ведётся менеджером.
```

Документы рендерятся в **sandboxed iframe** (defense-in-depth против XSS).
Все пользовательские поля в шаблонах экранируются.

## 4. Что требует настройки для продакшена

Эти переменные окружения включают «боевые» режимы (в Render → Environment):

| Переменная | Блок | Без неё |
|---|---|---|
| `TURNSTILE_SITE_KEY` / `TURNSTILE_SECRET_KEY` | Регистрация | работают test-ключи (пропускают всех — реальной анти-бот защиты нет) |
| `KASPI_MERCHANT_ID` / `KASPI_API_TOKEN` | Оплата | mock-режим (кнопка «Я оплатил»), реальный webhook отключён |
| `RESEND_API_KEY` | Письма | ссылки восстановления логируются в Render Logs (не уходят на почту) |
| `OPENAI_API_KEY` | Чат | AI-перевод не выполняется (показывается оригинал) |
| `TELEGRAM_BOT_TOKEN` / `TELEGRAM_MODERATOR_CHAT_ID` | Поддержка | сообщения в Службу Поддержки не форвардятся в Telegram |
| `S3_*` (R2/Spaces) | Файлы | файлы пишутся в `/tmp/` и **стираются при каждом деплое** |
| `CORS_ORIGIN`, `JWT_ACCESS_SECRET`, `JWT_REFRESH_SECRET` | Безопасность | прод не стартует с небезопасной конфигурацией (assert при boot) |

## 5. Сводная схема (жизненный цикл участника)

```
        РЕГИСТРАЦИЯ                ДОСТУП                 ОПЛАТА                СВЯЗЬ
   ┌──────────────────┐    ┌──────────────────┐   ┌────────────────┐   ┌──────────────────┐
   │ выбор роли        │    │ анкета → проверка │   │ пакет токенов   │   │ чат + AI-перевод │
   │ + Turnstile       │──► │ → VERIFIED        │──►│ → Kaspi Pay QR  │──►│ Служба Поддержки │
   │ + click-wrap 2.0  │    │ доступ к Реестру  │   │ → токены        │   │ WhatsApp / TG /  │
   │ → JWT + cookie    │    │                   │   │ → открытие карт.│   │ email            │
   └──────────────────┘    └──────────────────┘   └────────────────┘   └──────────────────┘
                                                          │
                                                   крупная сделка
                                                          ▼
                                              Сопровождение: NDA→NCNDA→
                                              комиссия 1,5% (лимит $1,5 млн)
```
